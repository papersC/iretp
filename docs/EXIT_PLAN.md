# IRETP Knowledge Transfer & Exit Plan

**RFP reference:** DLD-IRETP-2026-001 v1.3, §16.2 deliverable #70 + §19.1 data-ownership clause
**Document purpose:** Describe how DLD receives full platform ownership at project completion and at any future contract expiry. Covers source code, IaC scripts, all documentation, infrastructure control, and data.

---

## 1. Ownership principles

Per RFP §19.1:

- **All data** processed, stored, or generated by IRETP is the sole property of DLD.
- **All custom code** written for IRETP vests in DLD on payment of the relevant milestone.
- **All documentation** in this repository is delivered to DLD as part of acceptance.
- **All IaC scripts** that provision the production environment are handed to DLD.
- Open-source dependencies retain their original licences (MIT / Apache-2.0 / BSD-3 — non-restrictive).

## 3. Knowledge transfer programme (Phase 4)

Per RFP Phase 4 deliverable #54, the vendor delivers **8 structured training sessions**:

| # | Session | Duration | Audience | Outcomes |
|---|---|---|---|---|
| 1 | Platform architecture overview | 90 min | DLD IT + DLD CISO | Each attendee can locate the 3 service tiers, the data layer, the AI tier, and the integration touchpoints |
| 2 | CMS authoring + publishing | 60 min | DLD content authors | Each attendee can edit, preview, publish, and roll back a CMS page without vendor help |
| 3 | EWRS operations | 90 min | DLD Operations + Supervisors | Each attendee can acknowledge an alert, run a playbook, escalate, and adjust a threshold |
| 4 | Developer Performance & Rating | 60 min | DLD Compliance team | Each attendee can read the leaderboard, compare developers, and explain composite-score weights |
| 5 | AI Agent administration | 60 min | SystemAdministrator role | Each attendee can switch active models, view the accuracy harness, and read the AI audit log |
| 6 | Open Data API portal | 45 min | DLD API liaison + DLD comms | Each attendee can issue an API key, set tier rate limits, and read the developer portal |
| 7 | Security incident response | 90 min | DLD CISO + Vendor on-call | Walk through SIRP playbook with the runbook open; tabletop a P1 incident |
| 8 | Microsoft Fabric / OneLake integration | 60 min | DLD Fabric data engineer + Vendor data lead | Each attendee can navigate the Architecture Integration Map and check freshness from `/admin/fabric` |

Each session includes:
- Pre-read (1 page) shared 48 h ahead
- Live demo
- Hands-on exercise on the staging environment
- Recorded video + slide deck handed over
- Optional 1-hr refresher session during the 12-month warranty period (4 are pre-paid per RFP §15.1)

## 4. Exit triggers & timeline

The exit plan executes when any of the following occur:

| Trigger | Timeline |
|---|---|
| Contract expiry at end of warranty year | 90 days notice → handover phase begins |
| DLD-initiated termination for convenience | 60 days notice → handover phase begins |
| DLD-initiated termination for cause | Immediate handover; vendor cooperates without prejudice |
| Vendor insolvency | Source escrow released immediately; DLD takes operational control |

### Handover phase (typical 60 days)

| Week | Activity | Deliverable |
|---|---|---|
| 1 | Joint handover kickoff with DLD-assigned successor team | Handover plan signed |
| 2 | Source repository transferred to DLD-controlled Git server | Repo URL handed over |
| 2 | IaC scripts replayed against a DLD-controlled subscription | Working replica in DLD environment |
| 3–4 | Pair-programming + shadowing sessions | Successor team can deploy + run tests independently |
| 5 | DR drill executed by successor team | Drill report |
| 6 | Production access transferred (Azure Portal owners, Key Vault, DNS) | Access matrix updated |
| 7 | Vendor exits production access; final monitoring period begins | Final monitoring report |
| 8 | Final data extract + data destruction certificate from vendor systems | Signed certificate |

## 5. Data return obligations

Per RFP §19.1, on contract expiration / termination / handover, the vendor must:

1. Deliver **all DLD data** in machine-readable, open formats within **30 calendar days**:
   - OneLake Gold tables as Apache Parquet
   - Operational data (alerts, watchlists, audit log) as `.bacpac` or SQL dump
   - Generated PDFs (escrow reports, investor profiles) as a single archived directory
   - All CMS content as exported JSON from the CMS provider
2. Provide **written certification of complete data destruction** from all vendor-controlled systems within **60 days** of handover completion.
3. Destroy any cached / replicated copies of DLD data held by sub-contractors (SMS / SMTP / monitoring providers).

The vendor maintains no rights to use DLD data for any purpose after handover.

## 6. Continuity safeguards

| Risk | Safeguard |
|---|---|
| Knowledge concentrated in one vendor engineer | Pair-programming + shared documentation maintained throughout delivery; no single point of human failure |
| IaC scripts depend on vendor-only modules | All Bicep modules are self-contained and runnable from a clean Azure subscription with only DLD credentials |
| CI/CD pipelines depend on vendor-only secrets | Production secrets are in DLD-owned Key Vault from Phase 1; vendor never holds copies |
| Documentation becomes stale | The compliance-matrix traceability test (`ComplianceMatrixTraceabilityTests.cs`) fails CI when a referenced file is moved without updating the doc — keeps documentation accurate as code changes |
| AI model provider lock-in | Multi-model orchestration layer per RFP §5.3 lets DLD swap providers without rebuilding the application |
