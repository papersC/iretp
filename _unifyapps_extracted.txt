[Heading 2] 4.2 Multi-Model AI Architecture
DLD's requirement for a multi-model AI architecture. with the ability to switch models from the administration interface without redeployment. is a precise description of how UnifyApps Agentic AI is architected. Our AI orchestration layer sits between the AI Agent and the underlying language models. It enables:
Different models for different task types: for example, a high-capability model for complex data retrieval and analysis queries, and a lighter, faster model for DLD service navigation. allocating inference cost to the queries that justify it. UnifyApps advises DLD on the optimal model selection per task type as part of our LLM evaluation process.
Model switching from the admin interface: the active model can be changed by a System Administrator without any code changes, redeployment, or vendor involvement
New model addition by configuration: adding a new AI model provider. whether a managed API service or a self-hosted open-source LLM. requires only a configuration update in the orchestration layer. No redeployment or vendor involvement needed.
[Heading 2] 4.3 RAG Architecture. Grounding Every Response in DLD Data
Retrieval-Augmented Generation is mandatory for every response the AI Agent delivers. This means the Agent never generates an answer from general training data alone. every factual claim is grounded in retrieved, timestamped DLD data. The RAG pipeline operates as follows:
Step 1. Query Parsing: The user's natural-language question is parsed and classified (data query, service navigation, analytical request, comparative query)
Step 2. Retrieval: OpenSearch retrieves the most relevant data chunks from the indexed Gold layer, including transaction records, project data, developer profiles, zone boundaries, RERA regulations, and DLD service procedures. OpenSearch provides the same vector search capability as Azure AI Search at zero additional licence cost. it is used identically on both the on-prem and Azure deployments
Step 3. Augmentation: Retrieved data is injected into the LLM context alongside the system prompt (which enforces the non-advisory constraint and cites source requirements)
Step 4. Generation: The LLM generates a response grounded exclusively in the retrieved data. Every data point is cited with its source and timestamp.
Step 5. Validation: A post-generation validation layer checks the response for any investment advisory language. Flagged responses are either reformulated or declined with an appropriate message.
[Heading 2] 4.4 LLM Selection. Platform-Native Evaluation
UnifyApps does not recommend a specific LLM for IRETP. We run the evaluation on the platform itself.
The Unify Agentic AI platform has built-in tools to run structured experiments across multiple LLM providers simultaneously, measured against DLD's actual data, actual query patterns, and actual requirements. The result is an evidence-based selection. DLD sees the numbers and makes the decision.
Every LLM vendor will tell DLD their model is the best choice for Arabic real estate data. UnifyApps does not make that claim. In Phase 0 we connect each candidate model to the orchestration layer and run the same evaluation harness across all of them against DLD's actual Gold layer data. Accuracy, latency, cost, and Arabic NLP quality are measured against DLD's real queries, not generic benchmarks. The model that wins the evaluation is the one DLD deploys.
The Phase 0 evaluation, delivered as a written report on Day 14, covers every dimension that matters for IRETP:
The evaluation report presents all results in a side-by-side table. UnifyApps provides a written recommendation based on the data. DLD reviews, asks questions, and makes the final selection. We then configure the orchestration layer for the selected model within one working day.
Because the platform is LLM-agnostic, switching models after go-live is a configuration change with no redeployment and no downtime. If DLD wants to re-run the evaluation in Year 2 against newer models, the same harness runs again. The platform cost per session is the same regardless of which model is active.

[Heading 2] 9.2 Pre-Publication Analytics Assessment Methodology
Before any data goes live on the External Portal, the platform must pass a structured Pre-Publication Analytics Assessment. This is the RFP's required analytics assessment methodology, and it is a hard go-live gate, the portal does not launch until it passes and is signed off in writing by DLD's designated data liaison.
The assessment covers three layers of analytical validation:
The first layer is raw data reconciliation. A random sample of transaction records is pulled from the Gold layer and compared against what the platform displays. Required match rate: 99.5% or above. Any field-level discrepancy is traced back to the pipeline and resolved before the gate can pass.
The second layer is derived metric validation. Every computed metric displayed on the portal; price per sqft averages, YoY percentage changes, zone-level aggregations, off-plan share ratios, median values, rental yield calculations; is independently recalculated from the raw Gold layer data and compared against the platform output. Zero tolerance for calculation errors. Trend direction indicators (up arrows, down arrows, percentage labels) are verified for correct direction and correct magnitude.
The third layer is edge case review. Statistical outliers across all metric types are identified, categorised as legitimate extreme values or data entry errors, and handled by agreement with DLD before go-live. Any outlier that would materially mislead a public investor is flagged and either annotated or suppressed pending DLD instruction.
The assessment is conducted jointly by the UnifyApps delivery team and DLD's data liaison. Results are documented in a Pre-Publication Analytics Assessment Report, which becomes a permanent DLD data governance artefact. DLD provides written sign-off before the platform goes live. If the assessment fails at any threshold, the Phase 1 go-live date moves; this gate does not bend.
[Heading 2] 9.3 Ongoing Data Accuracy Monitoring
DLD's Gold layer is the source of truth. If the Gold layer contains an error, the IRETP will reflect that error. our platform connects to it faithfully. Our responsibility is to ensure the platform displays exactly what the Gold layer contains, with no transformation errors and no pipeline-introduced discrepancies.
[Heading 1] 10. Mandatory Data Integration and Accuracy Case Study
Civil Service and Government Development Bureau; Government of Qatar
Engagement overview
UnifyApps delivered the Government Services Performance Experience Platform for the Civil Service and Government Development Bureau (CGB), a central government authority in Qatar responsible for overseeing service quality across all ministries and government entities. The platform was delivered in partnership with malomatia, a Microsoft Gold Partner and leading Qatari IT services company. The platform serves the highest levels of Qatar's government, providing real-time service performance intelligence to the Prime Minister, Deputy Ministers, and ministry leadership teams. It operates bilingually in Arabic and English.
The platform is directly analogous to the IRETP in scope and complexity: a public-sector analytics and AI platform integrating multiple siloed government data sources into a single trusted intelligence layer, serving multiple user personas with different access levels, with a mandatory accuracy validation requirement before any data reaches senior decision-makers.
The data integration challenge
CGB faced a data landscape that is structurally identical to the challenge DLD faces with the IRETP. Five separate data source systems operated independently with no common data model, inconsistent formats, varying update cadences, and unreliable data quality that had actively eroded leadership trust in the existing reports:
The core problem was not the absence of data. CGB had data. The problem was that each source had been built independently, used different field names for the same concepts, applied inconsistent classification logic, and contained records that conflicted with records in other sources about the same citizen interaction. Aggregating across these sources without resolving those inconsistencies would produce scores and insights that were actively misleading.
How UnifyApps resolved the data quality issues
UnifyApps implemented a Unified Integration Layer, a middleware architecture that connected all five source systems into a standardized unified data model before any data reached the analytics layer or the user interface. No source data was presented to users in its raw form. Every record passed through validation and transformation before entering the CGB Azure Data Lake Gold layer.
Three specific data quality mechanisms were applied:
LLM-driven data structuring and classification. Unstructured Arabic-language content from social media and call centre records was processed by an LLM pipeline that classified each record into sentiment categories (Positive, Negative, Neutral), extracted topics, and assigned relevance scores. The LLM handled Gulf Arabic dialect variation,  a significant data quality challenge that rule-based systems cannot address, because the same concept is expressed differently across Qatari, Saudi, Egyptian, and Levantine Arabic.
Human-in-the-loop validation dashboard. Recognising that LLM classification introduces bias and inaccuracies, particularly on ambiguous Gulf Arabic phrasing,  UnifyApps built a mandatory human verification layer. CGB administrators reviewed a queue of AI-generated classifications before they were promoted to the production dataset. Every AI insight carried a confidence score. Reviewers could examine the underlying source record (the original social media post, service centre case, or call record) to verify the classification before approving it. This was not an optional QA step, no AI-classified record entered production without human sign-off.
Sentiment validation with linguistic expertise. Arabic-speaking subject matter experts from malomatia reviewed the AI sentiment classifications on a structured sample basis. The validation framework calculated Precision, Recall, and Weighted F1 Score per sentiment class. The F1 Score; the harmonic mean of precision and recall; was the primary KPI for data quality. Discrepancies between LLM output and human reviewer judgement were fed back into the system as a continuous improvement loop, improving classification accuracy over successive cycles.
Accuracy methodology and sign-off
The score calculation methodology used Analytical Hierarchical Process (AHP), a statistically rigorous weighting method, to derive each entity's Experience Score from the underlying source metrics. Every metric was normalised to a 0-100 scale before weighting, ensuring that metrics measured on different original scales (CSAT 0-5, transaction completion percentage, call resolution rate) contributed proportionally to the composite score.
Before any entity scores were presented to ministry leadership, UnifyApps ran a structured validation cycle:
The AI classification pipeline's Weighted F1 Score was confirmed above the agreed threshold across all three sentiment classes
A human reviewer sign-off queue cleared all pending validations
Score calculations were verified against the raw source data to confirm the derived weighted mean matched expected ranges
CGB administrators reviewed the initial score outputs for each entity before the platform was presented to Prime Minister-level users
The platform went live only after this multi-stage validation was completed and formally accepted by CGB.
Direct relevance to the IRETP
Every element of this engagement maps directly to what DLD requires for the IRETP:
The Unified Integration Layer connecting five disparate government source systems is the same architectural pattern proposed for the IRETP in Section 3.2, where UnifyApps connects to DLD's OneLake Gold layer via the native Fabric connector and handles the data transformation and standardisation pipeline.
The human-in-the-loop validation dashboard, where no AI-processed data reaches end users without administrator review,  is the same methodology proposed for the IRETP Pre-Publication Analytics Assessment in Section 9.2. DLD's data liaison signs off on the reconciliation report before the portal goes live. The mechanism is the same.
The six-persona RBAC model (Prime Minister, Prime Minister's Team, Deputy Minister, Deputy Minister's Team, Services Owner, CGB Admin) is structurally identical to the six internal roles defined for the IRETP internal platform.
The bilingual Arabic/English requirement, the AHP-based composite scoring methodology, and the leadership dashboard design serving heads of state, all delivered successfully in a GCC government context,  directly evidence UnifyApps' capability to deliver the IRETP.

[Heading 2] 17.1 Beneficial Ownership Transparency
JLL's 2024 GRETI report identified beneficial ownership transparency as a key gap for the UAE. International institutional investors. particularly those subject to FATF compliance requirements. need to understand the corporate ownership structures of the developers they are considering. Without this visibility, due diligence is incomplete, and the UAE cannot reach the "Highly Transparent" tier on this specific indicator.
UnifyApps proposes the following implementation approach, structured to work within the constraints of publicly available information:
Developer ownership disclosure module: where DLD or RERA holds publicly available corporate ownership information for registered developers (consistent with UAE commercial registry data), this is displayed on the Public Developer Scorecard. beneficial owner name, ownership percentage, and UAE commercial registration number.
Integration with UAE commercial registry: UnifyApps Integrations can connect to the UAE Federal Authority for Identity and Citizenship (ICA) commercial registry API or equivalent data source authorised by DLD, providing structured, machine-readable ownership data.
Investor-facing disclosure statement: where beneficial ownership data is not available for a specific developer, a clear disclosure statement is displayed. "Beneficial ownership data not available in public records". rather than a blank field. Transparency about the absence of data is itself a transparency measure.
GRETI impact: this feature directly addresses the Governance sub-index's beneficial ownership transparency indicator, which JLL assessed as a significant gap in the UAE's 2024 rating.
[Heading 2] 17.2 Mortgage & Debt Market Transparency
The GRETI 2024 report identified debt market transparency as a significant gap for the UAE. Aggregate mortgage data is already recorded in DLD's transaction registry (mortgage transactions are a transaction type in the existing data model), but it is not currently presented in a form that gives institutional investors the debt market intelligence they use in other transparent markets.
UnifyApps proposes the following mortgage and debt market transparency module:
Mortgage activity dashboard: total mortgage financing registered per zone per period, trend chart (24-month history), mortgage-to-total-transaction ratio, MoM and YoY trend.
Average LTV ratio by zone: where LTV data is available from RERA escrow records or DLD mortgage records, zone-level average LTV with distribution histogram.
Mortgage volume trend: total AED value of mortgages registered, split by property type, zone, and transaction status (off-plan vs. ready). This is fully derivable from existing DLD transaction data with no new source system required.
Data source: all mortgage data sourced exclusively from registered DLD transaction records. the same Gold-layer data source as the rest of the portal. No additional integration or data collection is required for the aggregate mortgage data layer.
